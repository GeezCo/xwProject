# 献微系统 — 离线部署指南

> 适用场景：生产环境**完全离线**（无 Docker Hub、无 PyPI、无 HuggingFace），目标机为 ARM64（鲲鹏 920 等）。
>
> 在线场景请参考 `迁移部署指南.md`。

---

## 总览

```
┌────────────────────┐                    ┌────────────────────┐
│  构建机（联网）     │  scp / U盘搬运      │  生产机（离线）    │
│  Mac arm64 / x86   │ ────────────────►  │  鲲鹏 aarch64       │
│                    │                    │                    │
│  • build-offline.sh│                    │  • setup.sh        │
│  • 拉镜像 + save tar│                    │  • docker load     │
│  • 下载模型权重     │                    │  • docker compose up│
└────────────────────┘                    └────────────────────┘
```

**关键约束**：构建机的 CPU 架构必须与生产机一致（都是 arm64 或都是 x86_64）。
否则镜像里的二进制不能运行。Mac M 系列和鲲鹏均为 arm64，**架构相同 → 无需交叉编译**。

## Part A — 构建机操作（联网）

### A1. 前置条件

| 工具 | 用途 |
|---|---|
| Docker 20.10+ | 拉镜像、构建镜像 |
| `huggingface-cli` | 下载 Embedding 模型 |
| 磁盘空间 | ≥10 GB |

```bash
pip install huggingface_hub                # 安装 hf 工具
export HF_ENDPOINT=https://hf-mirror.com   # 国内加速（推荐）
```

### A2. 准备 deploy 目录的可变内容

```bash
cd <项目根>

# 1) 编译后端 jar
cd xwBackend && ./mvnw clean package -DskipTests
cp target/uygur-project-0.0.1-SNAPSHOT.jar ../deploy/

# 2) 构建前端 dist
cd ../xwFrontend && npm install && npm run build
cp -r dist ../deploy/

# 3) 数据库导出（如要带数据）
mysqldump -h <旧IP> -P <端口> -u root -p<密码> \
    --databases uygur_project \
    --single-transaction --quick \
    > ../deploy/init.sql
```

### A3. 编辑 `.env`

打开 `deploy/.env`，确认：

```ini
# 端口（按现场实际网络规划改）
FRONTEND_PORT=9201
BACKEND_PORT=9202
ALGORITHM_PORT=9203
MYSQL_PORT=9204
MINIO_API_PORT=9205
MINIO_CONSOLE_PORT=9206
ES_PORT=9207            # ★ 新增
EMBEDDING_PORT=9208     # ★ 新增

# 服务器 IP
SERVER_IP=<生产机 IP>

# LLM 配置（占位变量，注入算法容器，但算法代码当前不读这些 env）
# 真正生效的 LLM 配置在 算法/数据抽取-新/config.json
LLM_BASE_URL=http://<现场LLM IP>:<端口>/v1
LLM_MODEL=Qwen/Qwen3.5-35B-A3B-w8a8-mtp
LLM_API_KEY=EMPTY
```

> **⚠️ LLM 配置的真正生效位置**
>
> 系统架构中后端**不直接调用 LLM**——所有 LLM 请求都由「算法服务」转发。
> 算法服务的 LLM 配置在：
>
> ```
> deploy/算法/数据抽取-新/config.json
> ```
>
> 生产现场修改 LLM 时，编辑此文件后重启算法容器：
> ```bash
> docker compose restart algorithm
> ```
>
> `.env` 中的 `LLM_BASE_URL` 等仅作占位，**当前算法代码不读这些环境变量**。

### A4. 一键打包

```bash
cd deploy
bash build-offline.sh
```

脚本会自动完成：
1. `docker pull --platform linux/arm64` 拉取 5 个公共镜像
2. `docker build` 出 `xianwei-embedding:offline`
3. 下载 `BAAI/bge-large-zh-v1.5` 模型到 `deploy/models/bge-large-zh-v1.5/`
4. `docker save` 全部镜像到 `deploy/offline-images.tar`

预计输出：
- `offline-images.tar` 约 **3-4 GB**
- `models/bge-large-zh-v1.5/` 约 **1.3 GB**

### A5. 整体打包搬运

```bash
cd <项目根>
tar czf xianwei-deploy.tar.gz deploy/
# 通过 U 盘 / scp / 移动硬盘转移到生产机
```

## Part B — 生产机操作（离线）

### B1. 前置条件

| 项目 | 要求 |
|---|---|
| 操作系统 | Linux (CentOS 7+ / Kylin / Ubuntu 20.04+) |
| 架构 | aarch64（必须与构建机一致） |
| Docker | 20.10+，**已离线安装好** |
| Java | JDK 8（已安装 / `deploy/jdk8*.tar.gz` 解压即用） |
| 磁盘 | ≥20 GB |
| 内存 | ≥8 GB（ES 至少 1G heap + Backend 2G + 其他） |

### B2. 上传并解包

```bash
# 假设上传到 /home/yinbanghu/
cd /home/yinbanghu
tar xzf xianwei-deploy.tar.gz
cd xwProject/  # 或 xianwei/，看你目录结构
```

### B3. 一键部署

```bash
bash deploy/setup.sh
```

脚本会自动：
1. 检查 `deploy/` 必备文件
2. 分发文件到运行时目录
3. **`docker load -i offline-images.tar`** 一次性加载所有镜像
4. 启动 MySQL → MinIO → ES（首次约 60s）
5. 启动 Algorithm → Embedding（首次加载模型约 2-3 分钟）→ Frontend
6. 启动后端 jar（screen 进程）
7. 健康检查 7 个服务

成功输出：
```
[OK]   前端 Nginx (:9201)
[OK]   后端 Spring Boot (:9202)
[OK]   算法 Flask (:9203)
[OK]   MinIO (:9205)
[OK]   Elasticsearch (:9207)
[OK]   Embedding (:9208)
[OK]   MySQL (:9204)
============================================
 部署完成  7/7 服务正常
============================================
```

### B4. RAG 端到端冒烟

```bash
source .env

# 1) 触发索引（取近 7 天报文）

    -H "Content-Type: application/json" \
    -d "{\"startDate\":\"$(date -d '7 days ago' +%Y-%m-%d)\",\"endDate\":\"$(date +%Y-%m-%d)\"}"

# 2) 查索引状态

    -H "Content-Type: application/json" \
```

## 常见问题

### Q1：`docker load` 报 `no matching manifest for linux/arm64`？

构建机和生产机架构不一致。检查：
```bash
uname -m  # 构建机和生产机都执行，必须输出相同（aarch64 / arm64）
```

### Q2：Embedding 容器启动慢？

首次加载模型约 2-3 分钟，正常。`docker logs xianwei-embedding -f` 看进度。
如超过 5 分钟，检查 `models/bge-large-zh-v1.5/` 是否完整（必须有 `pytorch_model.bin` 或 `model.safetensors`）。

### Q3：ES 启动后立刻 OOM？

调整 `.env` 中 `ES_JAVA_OPTS`：
```ini
ES_JAVA_OPTS=-Xms512m -Xmx512m   # 内存紧张时
```

### Q4：算法服务一直调不通 LLM？

`.env` 中的 `LLM_BASE_URL` 必须是**生产现场内网可达**的地址。
试一下：
```bash
curl http://<LLM_IP>:<端口>/v1/models
```

### Q5：现场没有 huggingface-cli 怎么下模型？

模型权重可以**任意机器**下载后拷贝。文件清单：
```
bge-large-zh-v1.5/
├── config.json
├── pytorch_model.bin         (~1.3 GB)
├── model.safetensors         (二选一)
├── tokenizer.json
├── tokenizer_config.json
├── special_tokens_map.json
└── vocab.txt
```
