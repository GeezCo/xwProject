1.1 标识

　　本文档适用于献微系统的软件需求规格说明。文档标识号：XW-REQ-001，版本号：V1.0.0。本文档针对的计算机软件配置项（CSCI）为献微系统软件，CSCI标识号为XW-CSCI-001。

1.2 系统概述

　　献微系统是一个基于大语言模型（LLM）的军事情报文章撰写与数据分析系统。系统采用前后端分离的三层架构（Vue 3前端 + Spring Boot后端 + Flask算法服务），通过算法服务调用大语言模型，实现智能化的事件抽取和报文融合功能。

　　系统的总体架构如下：

```
┌─────────────────┐     HTTP      ┌─────────────────┐     HTTP      ┌─────────────────┐
│    前端层       │ ───────────▶ │    后端层       │ ───────────▶ │   算法服务层    │
│  Vue 3 + Vite   │              │  Spring Boot    │              │  Flask + LLM    │
│    :8080        │              │    :8081        │              │    :5001        │
└─────────────────┘              └─────────────────┘              └─────────────────┘
                                        │                                │
                                        ▼                                ▼
                                 ┌─────────────┐                ┌─────────────┐
                                 │  MySQL 8.0  │                │  外部服务   │
                                 │   :8512     │                │  LLM/MinIO  │
                                 └─────────────┘                └─────────────┘
```

图1-1 献微系统总体架构图

　　系统包含以下三个核心功能模块：

a) 报文管理模块：负责军事情报报文的查询、筛选、分类管理、详情查看和批量导入，支持文字报和图文报两种模态类型
b) 属性抽取模块：基于大语言模型对报文进行层次化事件抽取，提取时间、地点、主体、动作等关键信息，并通过RexUniNlu服务进行多标签分类
c) 报文融合模块：将多篇相关报文的抽取结果进行融合，通过LLM生成包含标题、摘要、时间线和详细内容的综合分析报告

3 需求

3.1 要求的状态和方式

　　CSCI应在以下状态下运行：

a) 正常运行状态：所有服务（前端、后端、算法服务、MySQL、MinIO）正常运行，系统提供全部功能
b) 维护状态：部分服务停止运行，进行系统升级、数据迁移或配置变更
c) 降级运行状态：当LLM服务不可用时，报文管理功能（查询、筛选、详情查看）仍应正常使用，属性抽取和报文融合功能返回明确的错误提示

　　CSCI应支持以下运行方式：

a) 开发模式：前端通过Vite开发服务器（:8080）运行，支持HMR热更新
b) 生产模式：前端构建为静态文件由Nginx提供服务，后端以JAR包方式运行，Docker Compose编排基础服务

3.2 CSCI能力需求

3.2.1 报文管理能力（REQ-TEXT）

　　系统应提供完整的军事情报报文管理功能，包括报文查询、分类管理、详情查看和数据导入。

3.2.1.1 REQ-TEXT-001 报文分页查询

a) 功能描述：系统应支持报文分页查询功能，用户可通过分类ID、模态类型和关键词进行筛选
b) 输入要求：
   1) pageNum：整数，页码，从1开始，必填
   2) pageSize：整数，每页条数，范围1至100，默认10
   3) typeId：整数，分类ID，可选
   4) modalType：字符串，报文模态类型（文字报/图文报），可选
   5) keyword：字符串，搜索关键词，可选
c) 输出要求：
   1) 查询成功：返回分页结果，包含total（总记录数）、list（报文列表，含sid、title、content、times、type、modalType、images、isExtracted）、pageNum、pageSize
   2) 查询失败：返回错误码（code=0）和错误信息
d) 性能要求：在1万条报文数据规模下，响应时间应不超过2秒
e) 合格性方法：测试

3.2.1.2 REQ-TEXT-002 报文详情查看

a) 功能描述：系统应支持按报文唯一标识查看报文详情，包括文本内容和图片
b) 输入要求：
   1) sid：长整型，报文唯一标识，必填
c) 输出要求：
   1) 返回完整的OriginTextDTO对象，包含sid、title、content、times、type、modalType、images、isExtracted
   2) 图文报应返回images字段（JSON数组格式的图片文件名列表）
d) 性能要求：响应时间应不超过1秒
e) 合格性方法：测试

3.2.1.3 REQ-TEXT-003 分类目录管理

a) 功能描述：系统应支持分类目录的树形展示、新增、删除和批量删除操作
b) 输入要求：
   1) 查询：无参数
   2) 新增：typeName（分类名称）、parentId（父分类ID，0表示一级分类）
   3) 删除：id（分类ID）
c) 输出要求：
   1) 查询：返回树形结构的分类目录列表（支持多级嵌套，通过parentId自关联实现）
   2) 新增/删除：返回操作结果
d) 约束条件：删除分类时应支持级联删除（同时删除分类下的报文）
e) 合格性方法：测试

3.2.1.4 REQ-TEXT-004 报文批量导入

a) 功能描述：系统应支持JSONL格式的报文批量导入功能
b) 输入要求：
   1) 上传文件：JSONL格式，每行一条JSON记录，包含title、content、times、type等字段
c) 输出要求：
   1) 导入成功：返回成功导入的记录数
   2) 导入失败：返回失败原因及失败记录信息
d) 约束条件：系统应支持UTF-8、GBK、GB2312编码的文件
e) 合格性方法：测试

3.2.1.5 REQ-TEXT-005 报文删除管理

a) 功能描述：系统应支持单条删除和批量删除报文功能
b) 输入要求：
   1) 单条删除：sid（报文唯一标识）
   2) 批量删除：sid列表
c) 输出要求：返回删除操作结果
d) 约束条件：删除报文应同时清除关联的抽取结果（extraction_result表中对应记录）
e) 合格性方法：测试

3.2.2 属性抽取能力（REQ-EXT）

　　系统应提供基于大语言模型的军事事件属性抽取功能，自动从报文中提取结构化的事件信息。

3.2.2.1 REQ-EXT-001 执行属性抽取

a) 功能描述：系统应支持对指定报文执行基于LLM的层次化事件抽取，提取事件的时间、地点、主体、动作和分类标签
b) 输入要求：
   1) originTextId：长整型，报文唯一标识，必填
   2) force：布尔型，是否强制重新抽取，默认false
c) 输出要求：
   1) 抽取成功：返回结构化事件列表，每个事件包含event_id（事件编号）、time（时间）、location（地点数组）、subject（主体数组）、action（动作描述）、labels（分类标签数组，取值范围：人物/组织/部队/军舰/火炮/战机/武器/装备）、original_text（原文句子）
   2) 抽取失败：返回错误码和错误信息
d) 性能要求：
   1) 首次抽取：响应时间应不超过5分钟
   2) 缓存命中（已抽取过的报文且force=false）：响应时间应不超过1秒
e) 约束条件：
   1) 抽取结果应持久化到数据库extraction_result表
   2) 抽取完成后应更新origin_text表的is_extracted状态
f) 合格性方法：测试

3.2.2.2 REQ-EXT-002 查询抽取结果

a) 功能描述：系统应支持按报文ID查询已保存的属性抽取结果
b) 输入要求：
   1) originTextId：长整型，报文唯一标识，必填
c) 输出要求：
   1) 存在结果：返回ExtractionResultDTO对象，包含originTextId、eventsJson（事件列表JSON）、labelsJson（标签列表JSON）、extractionTime（抽取时间）
   2) 不存在结果：返回空数据
d) 性能要求：响应时间应不超过1秒
e) 合格性方法：测试

3.2.3 报文融合能力（REQ-FUS）

　　系统应提供多篇报文的融合分析功能，通过整合多篇报文的抽取结果和LLM生成能力，产出综合分析报告。

3.2.3.1 REQ-FUS-001 创建融合报告

a) 功能描述：系统应支持选择多篇报文生成融合分析报告，报告包含标题、摘要、时间线、详细内容、实体列表和标签
b) 输入要求：
   1) reports：报文数据数组，至少2个元素，每个包含id、title、content
   2) fusionType：字符串，融合类型
   3) customTitle：字符串，自定义标题，可选
c) 输出要求：
   1) 融合成功：返回融合报告，包含fusionId（唯一ID）、title（标题）、summary（约200字摘要）、timeline（时间线数组）、content（Markdown格式详细内容）、entities（按类别分组的实体字典）、labels（标签列表）、sourceIds（源报文ID列表）
   2) 融合失败：返回错误码和错误信息
d) 性能要求：融合报告生成时间应不超过5分钟
e) 约束条件：源报文应已完成属性抽取
f) 合格性方法：测试

3.2.3.2 REQ-FUS-002 保存融合报告

a) 功能描述：系统应支持将生成的融合报告保存到数据库
b) 输入要求：
   1) FusionDTO对象：包含完整的融合报告数据
c) 输出要求：返回保存成功的确认信息，包含生成的fusionId
d) 性能要求：响应时间应不超过1秒
e) 合格性方法：测试

3.2.3.3 REQ-FUS-003 融合报告列表查询

a) 功能描述：系统应支持融合报告的分页列表查询
b) 输入要求：
   1) pageNum：整数，页码
   2) pageSize：整数，每页条数
c) 输出要求：返回分页结果，包含报告的基本信息（fusionId、title、summary、createTime）
d) 性能要求：响应时间应不超过2秒
e) 合格性方法：测试

3.2.3.4 REQ-FUS-004 融合报告导出

a) 功能描述：系统应支持将融合报告导出为PDF和Word两种格式
b) 输入要求：
   1) id：长整型，融合报告ID
c) 输出要求：
   1) 导出PDF：返回PDF格式的文件流
   2) 导出Word：返回DOCX格式的文件流
d) 性能要求：导出时间应不超过10秒
e) 合格性方法：测试

3.2.4 数据统计能力（REQ-STAT）

3.2.4.1 REQ-STAT-001 核心指标概览

a) 功能描述：系统应提供核心指标概览功能，展示报文总数、已抽取数、融合报告数、实体总数等关键指标
b) 输入要求：无
c) 输出要求：
   1) totalTexts：整数，报文总数

e) 合格性方法：测试

e) 合格性方法：测试

3.2.4.3 REQ-STAT-003 模态分布统计

a) 功能描述：系统应提供报文按模态类型（文字报/图文报）的分布统计功能
b) 输入要求：无
c) 输出要求：返回模态分布数组，每个元素包含modalType（模态类型）、count（报文数量）、percentage（占比）

e) 合格性方法：测试

3.2.4.4 REQ-STAT-004 时间分布统计

a) 功能描述：系统应提供报文按时间维度的分布统计功能，支持按日、周、月、年聚合
b) 输入要求：
   1) granularity：字符串，时间粒度（day/week/month/year），默认month
   2) startDate：日期，起始日期，可选

e) 合格性方法：测试

3.2.4.5 REQ-STAT-005 抽取进度统计

b) 输入要求：无

e) 合格性方法：测试

3.2.4.6 REQ-STAT-006 高频标签统计

e) 合格性方法：测试

e) 合格性方法：测试

3.2.4.8 REQ-STAT-008 近期融合报告统计

3.2.5 情报分析能力（REQ-INTEL）

　　系统应提供情报分析功能，包括知识图谱构建、事件时间轴分析和地理热力图展示。

3.2.5.1 REQ-INTEL-001 知识图谱构建

3.6 适应性需求

　　系统应满足以下适应性需求：

a) REQ-ADP-001：系统应支持在x86_64和ARM64两种CPU架构上部署运行
b) REQ-ADP-002：系统应支持通过修改配置文件（config.json）切换不同的LLM模型（如Qwen3.5-122B-A10B与GLM-5之间的切换），无需修改代码
c) REQ-ADP-003：系统应支持通过环境变量（.env文件）配置服务端口号，适应不同的部署环境
d) REQ-ADP-004：前端应通过动态获取MinIO配置（GET /uygur/config）适配不同的存储服务地址，无需重新构建

3.7 保密性（Security）需求

　　系统应满足以下保密性需求：

a) REQ-SEC-001：系统应采用token认证机制，未经认证的请求不应访问系统功能接口
b) REQ-SEC-002：前端应实现路由守卫，未持有有效token的用户应被重定向到登录页面
c) REQ-SEC-003：后端应配置CORS白名单，仅允许指定来源域名的跨域请求
d) REQ-SEC-004：数据库应使用独立的访问账户（非root），限制操作权限范围

3.8 安全性（Safety）需求

　　系统应满足以下安全性需求：

a) REQ-SAF-001：LLM API调用失败时，系统应进行最多3次重试，所有重试均失败后应返回明确的错误提示，不应导致系统异常终止
b) REQ-SAF-002：数据库连接池耗尽时，系统应在60秒内返回错误提示，不应无限期阻塞
c) REQ-SAF-003：算法服务应实现日志轮转（单文件10MB，保留5份备份），防止日志文件无限增长耗尽磁盘空间

3.9 CSCI环境适应性需求

　　系统应满足以下环境适应性需求：

a) REQ-ENV-001：系统应支持Docker容器化部署，通过Docker Compose编排MySQL、MinIO、算法服务和Nginx容器
b) REQ-ENV-002：后端JAR包应支持通过screen进程管理运行，SSH断开后不应终止
c) REQ-ENV-003：系统应支持在无sudo权限的环境下部署运行

3.10 其他质量特性

3.10.1 可维护性

a) REQ-QA-001：系统应采用分层架构（Controller → Service → Mapper），各层职责明确，支持独立修改和测试
b) REQ-QA-002：后端应提供健康检查端点（/actuator/health），便于运维监控
c) REQ-QA-003：后端应集成Druid监控面板（/druid），提供SQL执行统计和慢查询分析

3.10.2 可用性

a) REQ-QA-004：前端报文列表页应实现keepAlive缓存，切换页面后返回时保持查询状态和滚动位置

6.1 术语说明

表6-1 术语说明表

| 术语 | 说明 |
|------|------|
| CSCI | 计算机软件配置项（Computer Software Configuration Item），指软件工程中可独立管理和配置的软件实体 |
| SRS | 软件需求规格说明（Software Requirements Specification），描述软件系统的功能、性能和约束需求的文档 |
| LLM | 大语言模型（Large Language Model），基于深度学习的自然语言处理模型，能够理解和生成文本 |
| REST | 表述性状态转移（Representational State Transfer），一种基于HTTP协议的Web服务架构风格 |
| CORS | 跨域资源共享（Cross-Origin Resource Sharing），浏览器安全机制，控制不同域名之间的HTTP请求 |
| JDBC | Java数据库连接（Java Database Connectivity），Java应用程序访问关系型数据库的标准接口 |
| MyBatis | Java持久层框架，提供SQL映射和对象关系映射功能，简化数据库操作 |

6.3 背景信息

　　献微系统是一个面向军事情报分析领域的智能辅助系统，旨在利用大语言模型的自然语言处理能力，提升军事情报报文的分析效率和质量。

　　系统的研发背景源于以下需求：

a) 军事情报报文数量大、内容复杂，人工分析耗时且易遗漏关键信息
b) 大语言模型在文本理解和生成方面的能力不断提升，具备了辅助情报分析的技术基础
c) 需要将分散的多篇情报报文进行关联融合，形成综合分析判断

　　系统采用前后端分离的三层架构设计，通过Vue 3前端提供用户交互界面，Spring Boot后端负责业务逻辑和数据管理，Flask算法服务封装大语言模型的调用逻辑。这种架构设计使各层可以独立开发、测试和部署，便于后续的功能扩展和技术升级。

　　系统的核心算法采用层次化抽取策略，首先通过规则筛选过滤无事件内容的段落，再对有事件特征的段落调用LLM进行精确抽取，有效降低了约50%的LLM调用成本。报文融合模块则通过整合多篇报文的抽取结果，利用LLM生成能力产出包含时间线分析和实体关联的综合报告。
