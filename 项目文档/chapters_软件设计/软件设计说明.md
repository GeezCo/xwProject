1 范围

1.1 标识

　　本文档适用于献微系统的软件设计说明。文档标识号：XW-SDD-001，版本号：V1.0.0。本文档针对的计算机软件配置项（CSCI）为献微系统软件，CSCI标识号为XW-CSCI-001。

1.2 系统概述

　　献微系统是一个基于大语言模型（LLM）的军事情报文章撰写与数据分析系统。系统采用前后端分离的三层架构，通过算法服务调用大语言模型，实现智能化的事件抽取、报文融合、情报分析、轨迹分析、告警管理和侦察筹划等功能。

　　系统的总体架构如下：

```
┌─────────────────┐     HTTP      ┌─────────────────┐     HTTP      ┌─────────────────┐
│    前端层       │ ───────────▶ │    后端层       │ ───────────▶ │   算法服务层    │
│  Vue 3 + Vite   │              │  Spring Boot    │              │  Flask + LLM    │
│    :8080        │              │    :8081        │              │    :5001        │
└─────────────────┘              └─────────────────┘              └─────────────────┘
                                        │                                │
                                        ▼                                ▼
                                 ┌─────────────┐                ┌─────────────┐
                                 │  MySQL 8.0  │                │  外部服务   │
                                 │   :8512     │                │  LLM/MinIO  │
                                 └─────────────┘                └─────────────┘
```

图1-1 献微系统总体架构图

　　系统主要由以下八个核心功能模块组成：

a) 报文管理模块：负责军事情报报文的查询、筛选、分类管理、详情查看和批量导入，支持文字报和图文报两种模态类型
b) 属性抽取模块：基于大语言模型对报文进行层次化事件抽取，提取时间、地点、主体、动作等关键信息，并通过RexUniNlu服务进行多标签分类
c) 报文融合模块：将多篇相关报文的抽取结果进行融合，通过LLM生成包含标题、摘要、时间线和详细内容的综合分析报告
d) 数据统计模块：提供报文、抽取、融合等多维度统计分析，包括数据概览、分类分布、时间分布、热门标签等
e) 情报分析模块：基于抽取结果构建知识图谱和事件时间线，支持实体关系查询、热力图展示和区域详情分析
f) 轨迹分析模块：对目标轨迹进行查询、预测、回放和异常检测，支持行军统计、集结地分析和路线分析
g) 告警管理模块：提供告警规则配置、实时告警推送和历史查询功能
h) 侦察筹划模块：支持情报缺口识别、侦察计划生成、资源调度和计划评估

3 系统级设计决策

3.1 输入输出设计决策

　　献微系统CSCI接收来自前端用户界面的HTTP请求作为输入，经过业务逻辑处理后以JSON格式输出响应结果。系统的主要输入输出设计决策如下：

3.1.1 输入设计

a) 报文管理输入：系统接收分页查询参数（pageNum、pageSize）、筛选条件（typeId分类ID、modalType模态类型、keyword关键词）作为报文列表查询的输入。所有输入通过HTTP POST请求以JSON格式提交。
b) 属性抽取输入：系统接收原始报文ID（originTextId）和强制抽取标志（force）作为属性抽取的输入。originTextId为长整型，force为布尔型，默认为false。
c) 报文融合输入：系统接收报文数据数组（reports[]）、融合类型（fusionType）和自定义标题（customTitle）作为融合报告生成的输入。每个报文数据包含id、title、content字段。
d) 数据统计输入：系统接收时间范围参数（days）、数量限制参数（limit）作为统计查询的输入，默认值分别为7天和20条。
e) 情报分析输入：系统接收实体类型过滤（entityTypes）、关系类型过滤（relationTypes）、时间范围（timeRange）、源报文ID数组（sourceIds）作为知识图谱构建和查询的输入。
f) 轨迹分析输入：系统接收目标ID（targetId）、时间范围（start、end）、预测时长（duration）、分辨率（resolution）作为轨迹查询和预测的输入。
g) 告警管理输入：系统接收告警规则JSON（包含规则类型、目标类型、触发条件、告警级别、通知方式）、告警处理动作（action：approve/reject/return）作为告警配置和处理的输入。
h) 侦察筹划输入：系统接收战区（theater）、侦察目的（purpose）、优先级（priority）、方案内容JSON（五步骤数据）、资源ID（asset_id）、调度时间（start_time、end_time）作为侦察方案编制和资源调度的输入。

3.1.2 输出设计

a) 统一响应格式：所有接口采用统一的ResultVO响应格式，包含code（状态码，1表示成功）、data（数据体）、msg（提示信息）三个字段
b) 分页响应格式：列表查询接口返回PageResultDTO格式，包含total（总记录数）、list（数据列表）、pageNum（当前页码）、pageSize（每页条数）
c) 抽取结果输出：属性抽取接口返回events_json（事件列表JSON）和labels_json（标签列表JSON），事件数据包含event_id、time、location、subject、action、labels、original_text字段
d) 融合报告输出：报文融合接口返回FusionDTO格式，包含fusionId、title、summary、timeline（时间线JSON）、content（Markdown格式详细内容）、entities（实体JSON）、labels（标签列表）
e) 统计数据输出：数据统计接口返回聚合统计结果，包含核心指标（totalTexts、extractedTexts、fusionReports、totalLabels）、分布数据（分类分布、模态分布、时间分布）、排行数据（高频标签TOP N、最近融合报告）
f) 知识图谱输出：情报分析接口返回图谱数据，包含nodes（节点数组，含entity_id、label、type、count）、edges（边数组，含source、target、type、weight）
g) 轨迹数据输出：轨迹分析接口返回目标详情（targetId、name、type、side、status、last_lat、last_lng）、历史轨迹点数组（lat、lng、altitude、speed、heading、recorded_at）、预测路线点数组（lat、lng、confidence、predicted_at）
h) 告警数据输出：告警管理接口返回告警规则列表、实时告警列表（rule_id、target_id、alert_level、content、status、triggered_at）、告警统计数据（按级别/状态分组统计）
i) 侦察方案输出：侦察筹划接口返回方案详情（plan_id、name、theater、purpose、priority、status、plan_content、scores）、资源列表（asset_id、name、type、status、capabilities）、调度任务列表（plan_id、asset_id、start_time、end_time、status）、评估结果（可行性/风险性/效率性/时效性四维度分数+改进建议）

3.2 行为设计决策

　　以下是CSCI响应各种输入条件和外部事件的行为设计决策：

3.2.1 SD-001 层次化抽取策略

a) 决策内容：属性抽取采用层次化抽取策略，第一层使用规则筛选（关键词匹配和句式特征识别），第二层对通过筛选的段落调用LLM进行精确抽取
b) 决策理由：
   1) 规则筛选可减少约50%的LLM调用，显著降低API调用成本和处理延迟
   2) 分段处理支持长文本抽取，不受LLM上下文窗口限制
   3) 合并去重机制保证最终结果的一致性和完整性
c) 相关需求：REQ-EXT-001（属性抽取）、REQ-ALG-001（事件抽取算法）
d) 影响范围：属性抽取模块（SU-002）、算法服务模块（SU-004）

3.2.2 SD-002 超时适配策略

a) 决策内容：前端HTTP请求超时设置为20分钟（1200000ms），后端RestTemplate连接超时和读取超时均设置为20分钟
b) 决策理由：
   1) 大语言模型处理复杂报文的抽取任务耗时约2至5分钟
   2) 报文融合涉及多篇报文的信息整合和LLM生成，耗时可能更长
   3) 网络波动可能导致LLM API响应延迟，需预留充足的超时时间
c) 相关需求：REQ-EXT-001、REQ-FUS-001
d) 影响范围：前端Axios配置（utils/request.js）、后端RestTemplate配置

3.2.3 SD-003 抽取结果缓存策略

a) 决策内容：已完成抽取的报文结果缓存于数据库extraction_result表中，后续请求默认直接返回缓存结果，仅在force=true时重新调用LLM
b) 决策理由：
   1) 同一篇报文的抽取结果具有幂等性，无需重复调用LLM
   2) LLM调用成本较高，缓存可避免不必要的资源消耗
   3) 通过origin_text表的is_extracted字段快速判断缓存状态
c) 相关需求：REQ-EXT-001、REQ-EXT-002
d) 影响范围：属性抽取模块（SU-002）

3.2.4 SD-004 三重JSON解析策略

a) 决策内容：对LLM返回的文本内容采用三重解析策略：策略1直接JSON解析，策略2提取Markdown代码块中的JSON，策略3使用花括号计数提取完整JSON对象
b) 决策理由：
   1) 不同LLM模型的输出格式不一致，部分模型会在JSON前后添加解释文字
   2) 部分模型使用Markdown代码块包裹JSON内容
   3) 三重策略逐级降级，最大化解析成功率
c) 相关需求：REQ-ALG-001、REQ-ALG-002
d) 影响范围：算法服务模块（SU-004）

3.2.5 SD-005 多模型兼容策略

a) 决策内容：算法服务兼容Qwen3.5-122B-A10B和GLM-5两种大语言模型，对GLM-5的reasoning_content字段进行特殊处理
b) 决策理由：
   1) 不同场景可能需要切换不同的LLM模型
   2) GLM-5模型返回内容在reasoning_content字段而非标准的content字段
   3) 通过配置文件config.json统一管理模型参数，支持热切换
c) 相关需求：REQ-ALG-001
d) 影响范围：算法服务模块（SU-004）

3.2.6 SD-006 LLM调用重试机制

a) 决策内容：对LLM API调用采用最多3次重试机制，每次重试间隔2秒
b) 决策理由：
   1) LLM API服务可能因网络波动或服务器负载出现临时性故障
   2) 重试机制可提高系统的容错能力和可用性
   3) 限制最大重试次数和间隔时间，避免无限等待
c) 相关需求：REQ-ALG-001、REQ-ALG-002
d) 影响范围：算法服务模块（SU-004）

3.3 数据库呈现设计决策

3.3.1 数据库选型

a) 决策内容：采用MySQL 8.0作为关系型数据库管理系统
b) 决策理由：
   1) MySQL 8.0支持JSON函数（JSON_EXTRACT、JSON_TABLE），适合存储事件抽取和融合报告中的复杂结构化数据
   2) MySQL生态成熟，与Spring Boot和MyBatis框架集成良好
   3) 支持Docker容器化部署，便于迁移和运维

3.3.2 数据表设计

a) 决策内容：设计16张数据表，包含4张现有核心表（origin_text、text_type、extraction_result、fusion_report）和12张新增表（intelligence_entity、intelligence_relation、intelligence_event、track_target、track_point、alert_rule、alert_record、recon_plan、recon_asset、recon_schedule、recon_approval、recon_result）
b) 决策理由：
   1) 原始报文与抽取结果分表存储，通过origin_text_id外键关联，支持按需抽取和增量处理
   2) 分类目录采用parent_id自关联实现树形结构，支持多级分类
   3) 复杂结构数据（事件列表、标签列表、时间线、实体列表）以JSON字符串存储在TEXT字段中，兼顾灵活性和查询性能
   4) 情报分析模块采用图数据库设计思想，intelligence_entity存储节点，intelligence_relation存储边，intelligence_event存储事件时间轴
   5) 轨迹分析模块采用时序数据设计，track_point表按target_id和recorded_at建立复合索引，支持高效的时间范围查询
   6) 告警管理模块分离规则配置（alert_rule）和告警记录（alert_record），支持规则复用和历史查询
   7) 侦察筹划模块采用主从表设计，recon_plan为主表，recon_schedule、recon_approval、recon_result为从表，通过plan_id关联

3.3.3 索引设计

a) 决策内容：在高频查询字段建立索引，包括origin_text表的type、modal_type、is_extracted字段，extraction_result表的origin_text_id字段，track_point表的(target_id, recorded_at)复合索引，alert_record表的rule_id、target_id、triggered_at字段，recon_plan表的status、theater字段
b) 决策理由：
   1) type和modal_type字段频繁用于筛选查询，索引可显著提升查询性能
   2) is_extracted字段用于快速判断抽取状态
   3) origin_text_id唯一索引保证每篇报文仅有一条抽取结果
   4) track_point表的复合索引支持按目标ID和时间范围的高效查询
   5) alert_record表的索引支持按规则、目标、时间的多维度查询
   6) recon_plan表的索引支持按状态和战区的快速筛选

3.4 安全保密设计决策

3.4.1 前端认证机制

a) 决策内容：前端采用token认证机制，用户登录后获取token并存储于本地，后续请求通过Axios拦截器自动注入token
b) 决策理由：
   1) 基于token的无状态认证适合前后端分离架构
   2) 路由守卫机制确保未认证用户无法访问系统功能页面

3.4.2 跨域安全控制

a) 决策内容：后端通过CorsConfig配置类限定允许的跨域来源（http://localhost:8080），仅允许GET、POST、PUT、DELETE方法，允许携带凭证
b) 决策理由：
   1) 前后端分离架构需要处理跨域请求
   2) 限定来源域名防止未授权的第三方站点调用后端接口

3.4.3 数据库访问安全

a) 决策内容：通过Druid连接池管理数据库连接，配置独立的数据库用户（uygur_user）和密码，启用Druid监控功能（/druid端点）
b) 决策理由：
   1) 连接池管理避免频繁创建和销毁数据库连接
   2) 独立用户权限限制数据库操作范围
   3) Druid监控面板提供SQL执行统计和慢查询分析

3.5 其他设计决策

3.5.1 SD-007 前端路由缓存

a) 决策内容：对searchResult（报文列表页）组件启用keepAlive缓存，避免切换页面后丢失查询状态和滚动位置
b) 决策理由：用户频繁在报文列表页与其他页面之间切换，缓存组件状态可提升使用体验

3.5.2 SD-008 多编码文件读取

a) 决策内容：算法服务在读取文本文件时依次尝试UTF-8、GBK、GB2312、UTF-16编码
b) 决策理由：批量导入的报文文件来源多样，编码格式不统一，多编码降级策略可提高文件读取的兼容性

3.5.3 SD-009 MinIO动态配置

a) 决策内容：前端启动时通过GET /uygur/config接口获取MinIO地址前缀（minioPrefix），图片URL通过拼接minioPrefix和文件名动态生成
b) 决策理由：MinIO服务地址可能因部署环境变化而不同，动态获取配置避免前端硬编码地址，迁移部署时无需重新构建前端

3.5.4 SD-010 WebSocket实时推送

a) 决策内容：对轨迹监控、告警通知、侦察任务监控采用WebSocket协议实现服务端主动推送，使用STOMP协议+SockJS兼容方案
b) 决策理由：
   1) 轨迹监控需要每15秒推送目标位置更新，HTTP轮询会产生大量无效请求
   2) 告警事件触发时需要立即通知前端，WebSocket可实现毫秒级推送延迟
   3) STOMP协议提供订阅/发布模式，支持多客户端分组推送
c) 相关需求：REQ-TRACK-001、REQ-ALERT-006、REQ-RECON-015
d) 影响范围：轨迹分析模块（SU-005）、告警管理模块（SU-006）、侦察筹划模块（SU-007）

3.5.5 SD-011 知识图谱构建策略

a) 决策内容：情报分析模块的知识图谱采用离线构建+增量更新策略，通过算法服务调用LLM进行实体消歧和关系推断，构建结果缓存于intelligence_entity和intelligence_relation表
b) 决策理由：
   1) 实体消歧和关系推断需要调用LLM，实时构建耗时过长
   2) 离线构建可批量处理多篇报文的抽取结果，提高构建效率
   3) 增量更新策略支持新报文抽取后自动更新图谱
c) 相关需求：REQ-INTEL-001、REQ-ALG-004
d) 影响范围：情报分析模块（SU-004）、算法服务模块（SU-008）

3.5.6 SD-012 轨迹预测算法选型

a) 决策内容：轨迹预测采用卡尔曼滤波算法，对历史轨迹点进行状态估计和预测，输出未来路线点及置信度
b) 决策理由：
   1) 卡尔曼滤波适合处理带噪声的时序数据，对军事目标轨迹预测效果良好
   2) 算法计算复杂度低，可实时响应前端预测请求
   3) 相比LSTM深度学习模型，卡尔曼滤波无需大量训练数据
c) 相关需求：REQ-TRACK-004、REQ-ALG-006
d) 影响范围：算法服务模块（SU-008）

3.5.7 SD-013 异常行为检测策略

a) 决策内容：轨迹异常检测采用Isolation Forest算法，基于目标历史行为基线识别偏离常规模式的异常事件
b) 决策理由：
   1) Isolation Forest无监督学习算法，无需标注异常样本
   2) 算法对高维特征（速度、方向、停留时间、区域等）的异常检测效果好
   3) scikit-learn库提供成熟实现，开发成本低
c) 相关需求：REQ-TRACK-012、REQ-ALG-010
d) 影响范围：算法服务模块（SU-008）

3.5.8 SD-014 侦察方案AI辅助生成

a) 决策内容：侦察方案编制采用LLM辅助生成策略，基于任务定义、资源清单、约束条件调用LLM生成五步骤方案内容（任务定义、情况分析、行动方案、保障措施、指挥协同）
b) 决策理由：
   1) 侦察方案编制需要综合考虑多种因素，人工编制耗时长
   2) LLM可基于模板和约束条件快速生成结构化方案文本
   3) AI生成的方案可作为初稿，由用户进一步修改完善
c) 相关需求：REQ-RECON-008、REQ-ALG-013
d) 影响范围：侦察筹划模块（SU-007）、算法服务模块（SU-008）

3.5.9 SD-015 方案评估多维度打分

a) 决策内容：侦察方案评估采用LLM+规则引擎混合策略，从可行性、风险性、效率性、时效性四个维度进行打分（0-100分），并生成改进建议
b) 决策理由：
   1) 可行性和风险性评估需要LLM理解方案内容和约束条件
   2) 效率性和时效性可通过规则引擎计算（资源利用率、时间窗口匹配度）
   3) 多维度打分为审批决策提供量化依据
c) 相关需求：REQ-RECON-017、REQ-ALG-014
d) 影响范围：侦察筹划模块（SU-007）、算法服务模块（SU-008）

4.2 执行方案

4.2.1 启动执行顺序

　　系统各软件单元的启动顺序如下：

a) 第一步：启动算法服务模块（SU-004），Flask应用监听5001端口。算法服务必须先于后端启动，因为后端启动后可能立即接收到抽取或融合请求。
b) 第二步：启动后端服务（SU-001至SU-003），Spring Boot应用监听8081端口。后端启动时初始化数据库连接池、RestTemplate和MinIO客户端。
c) 第三步：启动前端服务，Vite开发服务器监听8080端口（或生产环境由Nginx提供静态文件服务）。

4.2.3 异常处理

a) LLM API调用失败：算法服务采用最多3次重试机制，每次间隔2秒。3次重试均失败后返回错误响应
b) 数据库连接异常：Druid连接池配置最大等待时间60秒，超时后抛出异常，后端返回错误响应
c) 算法服务不可用：后端RestTemplate设置20分钟超时，超时后返回错误响应提示算法服务不可达
d) JSON解析失败：算法服务采用三重解析策略（SD-004），所有策略均失败时返回解析错误

4.2.4 并发执行

a) 后端Spring Boot应用支持多线程并发处理请求，Tomcat默认最大线程数200
b) 数据库连接池（Druid）配置最大活跃连接20，最小空闲连接5
c) 多个抽取请求可并发执行，每个请求独立调用算法服务
d) extraction_result表通过origin_text_id唯一索引保证同一报文不会产生重复抽取结果

5.3.5 逻辑流程

a) 执行启动条件：
   1) Spring Boot应用启动完成
   2) 算法服务（:5001）可达
   3) 源报文均已完成属性抽取

b) 操作顺序（创建融合报告）：
   1) 接收前端POST请求（/api/fusion/create），解析FusionCreateRequest参数
   2) 遍历reports数组，对每篇报文查询其抽取结果（ExtractionMapper.selectByOriginTextId）
   3) 组装请求数据：每篇报文包含id、title、content和extractionResult
   4) 通过RestTemplate发送POST请求到http://localhost:5001/fusion/create
   5) 等待算法服务返回融合报告
   6) 解析响应，返回融合报告JSON（此时未持久化）

c) 操作顺序（保存融合报告）：
   1) 接收前端POST请求（/api/fusion/save），解析FusionDTO参数
   2) 生成fusionId（时间戳）
   3) 调用FusionMapper.insertReport()将报告写入fusion_report表
   4) 返回保存成功响应

d) 控制传递条件：
   1) 融合成功 → 返回ResultVO(code=1, data=融合报告)
   2) 源报文不足 → 返回ResultVO(code=0, msg="至少需要2篇报文")
   3) 算法服务异常 → 返回ResultVO(code=0, msg="融合失败")

e) 输入响应时间：
   1) 创建融合报告：1至3分钟（取决于报文数量和LLM响应速度）
   2) 保存报告：小于200ms
   3) 列表查询：小于200ms
   4) 导出Markdown：小于1秒

5.4.3 SU-004-2 FusionExtractor（融合算法）

a) 子单元用途：将多篇相关报文的抽取结果进行融合，通过信息整合和LLM生成产出综合分析报告
b) 设计决策：
   1) 融合过程分为代码整合和LLM生成两个阶段，代码层面完成时间线合并、实体聚合和标签合并，LLM负责生成标题、摘要和详细内容
   2) 时间线按时间排序并去重
   3) 实体按类别分组聚合并去重
   4) LLM生成温度参数设为0.3，略高于抽取（0.1），允许更多创造性
c) 输入数据元素：
   1) reports：Array，源报文数据数组，每个报文包含id、title、content和extractionResult
   2) fusionType：String，融合类型
d) 输出数据元素：
   1) fusionId：Long，融合报告唯一ID（毫秒级时间戳）
   2) title：String，融合报告标题
   3) summary：String，200字左右的摘要
   4) timeline：Array，时间线列表，每项含time和description
   5) content：String，详细内容（Markdown格式，包含背景、事件经过、影响分析）
   6) entities：Object，按类别分组的实体字典
   7) labels：Array，标签列表
   8) sourceIds：Array，源报文ID列表
e) 逻辑流程：
   1) 数据预处理：遍历源报文，解析每篇报文的extractionResult中的eventsJson和labelsJson
   2) 信息整合（代码层面）：
      ① 合并时间线：从所有事件中提取time和action，构建时间点描述
      ② 合并实体：按类别（人物、组织、部队等）分组，使用Set去重
      ③ 合并标签：收集所有报文的标签，使用Set去重
   3) 时间线排序：按时间字段排序
   4) LLM生成：构建Prompt，将源报文摘要提供给LLM，要求生成标题、摘要和详细内容（Markdown格式）
   5) 调用LLM API（temperature=0.3，max_tokens=8000）
   6) 解析LLM响应，提取title、summary和content
   7) 结果组装：将代码整合的timeline、entities、labels与LLM生成的title、summary、content合并，添加fusionId和sourceIds
   8) 返回完整融合报告

7.3 背景信息

　　献微系统是为满足军事情报分析需求而开发的智能化信息处理系统。系统的核心创新点在于将大语言模型技术应用于军事报文的自动化事件抽取和多源信息融合分析领域。

　　系统采用层次化抽取策略（规则筛选 + LLM精确抽取），在保证抽取准确率的前提下将LLM调用成本降低约50%。该策略已在6000余篇报文数据集上完成验证。

　　系统支持多种大语言模型（Qwen3.5-122B-A10B、GLM-5等），通过OpenAI兼容API接口统一调用，模型切换仅需修改配置文件无需更改代码。系统已在超算平台（Paratera）和本地服务器（910B）两种环境下部署运行。

7.4 技术选型说明

a) 前端技术栈选型理由：
   1) Vue 3：采用Composition API提供更好的逻辑复用和TypeScript支持
   2) Vite：相比Webpack提供更快的冷启动和热更新速度
   3) Element Plus：成熟的Vue 3组件库，提供丰富的企业级UI组件
   4) Pinia：Vue 3官方推荐的状态管理方案，API更简洁

b) 后端技术栈选型理由：
   1) Spring Boot 2.7.18：稳定的LTS版本，生态成熟
   2) MyBatis：灵活的SQL映射框架，适合复杂查询场景
   3) Druid：提供SQL监控和连接池管理功能
   4) PageHelper：简化分页查询实现

c) 算法服务技术栈选型理由：
   1) Python Flask：轻量级Web框架，适合算法服务快速开发
   2) OpenAI SDK：统一的LLM调用接口，支持多模型切换
   3) 异步处理：采用同步HTTP接口配合后端20分钟超时，简化架构

d) 数据库选型理由：
   1) MySQL 8.0：成熟稳定，支持JSON字段存储复杂结构
   2) JSON字段：存储events_json、labels_json等非结构化数据，避免多表关联

e) 存储服务选型理由：
   1) MinIO：开源对象存储，兼容S3 API，适合图片存储

7.5 设计约束

a) 性能约束：
   1) 单篇报文抽取耗时：2-5分钟（受LLM推理速度限制）
   2) 报文融合耗时：3-8分钟（取决于源报文数量）
   3) HTTP超时设置：20分钟（适配LLM处理耗时）
   4) 数据库连接池：初始5个连接，最大20个连接

b) 安全约束：
   1) 前端token认证：所有API请求需携带有效token
   2) CORS白名单：仅允许配置的前端域名跨域访问
   3) SQL注入防护：使用MyBatis参数化查询
   4) 敏感信息保护：LLM API密钥存储于配置文件，不提交代码仓库

c) 兼容性约束：
   1) 浏览器支持：Chrome 90+、Edge 90+、Firefox 88+
   2) JDK版本：JDK 8或JDK 11
   3) Python版本：Python 3.10+
   4) MySQL版本：MySQL 8.0+

d) 部署约束：
   1) 启动顺序：算法服务 → 后端服务 → 前端服务
   2) 端口占用：8080（前端）、8081（后端）、5001（算法）、8512（MySQL）
   3) 网络依赖：需访问外部LLM API（Paratera平台）和RexUniNlu分类服务

7.6 未来扩展方向

a) 功能扩展：
   1) 数据统计模块：提供多维度统计分析和可视化
   2) 情报分析模块：构建知识图谱和事件时间线
   3) 轨迹分析模块：目标轨迹预测和异常检测
   4) 告警管理模块：实时告警推送和规则配置
   5) 侦察筹划模块：情报缺口识别和侦察方案生成
