# 献微系统数据库设计

## 概述

- **数据库**: MySQL 8.0
- **地址**: `36.103.234.242:8010`
- **库名**: `uygur_project`
- **字符集**: `utf8mb4`

---

## 一、表结构

### 1.1 origin_text — 原始文本表

存储导入的报文原文。

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| sid | INT | 自增主键 | 报文ID |
| title | VARCHAR(200) | - | 报文标题 |
| content | LONGTEXT | - | 报文正文内容 |
| times | VARCHAR(50) | - | 报文时间 |
| type | INT | - | 分类ID（关联 text_type.id） |
| modal_type | VARCHAR(20) | '文字报' | 模态类型：文字报/图文报/声像报 |
| is_extracted | TINYINT(1) | 0 | 是否已抽取：0-否，1-是 |
| is_indexed | TINYINT(1) | 0 | 是否已向量化入 ES：0-否，1-是（RAG 模块） |

**索引**:
- 主键: `sid`
- 普通索引: `idx_is_extracted (is_extracted)`
- 普通索引: `idx_is_indexed (is_indexed)`

---

### 1.2 text_type — 分类目录表

存储报文分类目录。

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| id | INT | 自增主键 | 分类ID |
| type_name | VARCHAR(50) | - | 分类名称（如"军事演习"、"武器装备"） |

---

### 1.3 extraction_result — 属性抽取结果表

存储每篇报文的LLM事件抽取结果。与 `origin_text` 一对一关系。

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| id | INT | 自增主键 | 主键 |
| origin_text_id | INT | - | 原始文本ID（关联 origin_text.sid） |
| extraction_time | DATETIME | CURRENT_TIMESTAMP | 抽取时间 |
| model | VARCHAR(50) | 'GLM-5' | 使用的LLM模型名称 |
| total_events | INT | 0 | 抽取事件总数 |
| labels_json | LONGTEXT | - | 分类标签JSON（如 `["部队","武器"]`） |
| entities_json | LONGTEXT | - | 实体JSON（如 `{"人物":[],"组织":["北约"],...}`） |
| events_json | LONGTEXT | - | 事件列表JSON |
| status | VARCHAR(20) | 'completed' | 状态：processing / completed / failed |
| error_message | TEXT | - | 错误信息（失败时记录） |

**索引**:
- 主键: `id`
- 唯一索引: `uk_origin_text_id (origin_text_id)`

**events_json 数据结构**:
```json
[
  {
    "event_id": 1,
    "time": "2024年3月15日",
    "location": ["横须贺", "关岛"],
    "subject": ["第七舰队", "萨德反导系统"],
    "action": "保持战备状态",
    "labels": ["部队", "武器"],
    "original_text": "原文句子..."
  }
]
```

**entities_json 数据结构**:
```json
{
  "人物": ["拜登"],
  "组织": ["北约", "联合国"],
  "部队": ["第七舰队"],
  "军舰": ["辽宁号"],
  "火炮": [],
  "战机": ["F-35"],
  "武器": ["萨德反导系统"],
  "装备": []
}
```

---

### 1.4 fusion_report — 融合报告表

存储多篇报文融合生成的综合报告。

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| id | BIGINT | 自增主键 | 融合报告ID |
| title | VARCHAR(200) | - | 报告标题 |
| summary | TEXT | - | 报告摘要 |
| timeline | JSON | - | 事件时间线 |
| content | TEXT | - | 详细内容（Markdown格式） |
| entities | JSON | - | 关键实体 |
| labels | JSON | - | 综合标签 |
| source_ids | VARCHAR(100) | - | 来源报文ID列表（JSON数组字符串） |
| model_used | VARCHAR(50) | - | 使用的LLM模型 |
| create_time | DATETIME | - | 创建时间 |
| update_time | DATETIME | - | 更新时间 |

**索引**:
- 主键: `id`

**timeline 数据结构**:
```json
[
  { "time": "2024-03-15", "description": "第七舰队在南海举行演习" },
  { "time": "2024-03-16", "description": "北约秘书长发表声明" }
]
```

---

## 二、表关系

```
text_type (1) ─────────── (N) origin_text
                                 │
                                 │ sid = origin_text_id
                                 │
                           (1) extraction_result

origin_text (N) ─ source_ids ─ (1) fusion_report
```

- `origin_text.type` → `text_type.id`（分类关联）
- `extraction_result.origin_text_id` → `origin_text.sid`（一对一，唯一索引）
- `fusion_report.source_ids` 存储参与融合的 `origin_text.sid` 列表（JSON数组，逻辑关联）

---

## 三、DDL脚本

所有DDL脚本位于 `后端/src/main/resources/db/` 目录：

| 文件 | 说明 |
|------|------|
| extraction_ddl.sql | 创建 extraction_result 表，添加 is_extracted 字段和索引 |
| fusion_report_ddl.sql | 创建 fusion_report 表 |
| origin_text_modal_type_ddl.sql | 为 origin_text 添加 modal_type 字段 |

**执行新DDL前注意**:
1. 先备份数据库
2. 在测试环境验证后再应用到生产
3. ALTER TABLE 新增字段应设默认值，避免影响现有数据
