# 代码修复日志 2026-05-03

## Bug: `_fix_json_format` 中文引号全局替换导致 JSON 解析错乱

### 影响范围

`算法/数据抽取-新/llm_event_extractor.py` 的 `LLMEventExtractor._fix_json_format()` 方法

### 问题现象

当原文包含中文引号（如 `"爱国者"`、`"里根"`）时，LLM 在 `original_text` 字段中保留了这些中文引号。解析阶段将所有中文引号全局替换为 ASCII 双引号后，字符串值内部出现了裸双引号，导致 JSON 结构被打断。

典型表现（以 index 71 为例）：

- LLM 实际返回了 2 个事件，解析后只剩 1 个
- `action` 字段被污染，吞掉了后续整段 JSON 文本
- `original_text` 包含残余的 JSON 片段

```
修复前 action 内容:
"保持战备状态",\n      "original_text": "2026 年 2 月 2 日...\"爱国者\"...驱逐舰保持战备状态。"\n    },\n    {\n      "time": "同日",\n      "location": ["南海"],\n...

修复后 action 内容:
"保持战备状态"
```

### 根因

原方法做了无差别的全局替换：

```python
# 修复前
def _fix_json_format(self, content: str) -> str:
    content = content.replace('“', '"').replace('”', '"')
    content = content.replace('：', ':')
    content = re.sub(r',\s*]', ']', content)
    content = re.sub(r',\s*}', '}', content)
    return content
```

`“`/`”` 替换不区分"JSON 结构引号"和"字符串值内容中的引号"。当 `original_text` 中出现 `"爱国者"` 时：

```
替换前（合法 JSON）:  "original_text": "...的“爱国者”防空系统..."
替换后（非法 JSON）:  "original_text": "...的"爱国者"防空系统..."
                                          ^         ^
                              被 JSON 解析器当作字符串边界，结构断裂
```

`json.loads()` 失败后走入兜底解析（花括号计数/正则），只捞出第一个事件对象的残骸，后续事件全部丢失。

### 修复措施

用状态机逐字符遍历，跟踪当前位置是否处于 JSON 字符串内部，只替换结构位置的中文引号：

```python
def _fix_json_format(self, content: str) -> str:
    CL = "“"  # Chinese left quote
    CR = "”"  # Chinese right quote
    FC = "："  # Fullwidth colon

    result = []
    in_string = False
    opened_by_chinese = False
    i = 0
    while i < len(content):
        ch = content[i]
        if in_string:
            if ch == "\\" and i + 1 < len(content):
                # 转义序列，跳过
                result.append(ch)
                result.append(content[i + 1])
                i += 2
                continue
            elif ch == '"':
                # 标准双引号关闭字符串
                in_string = False
                opened_by_chinese = False
                result.append(ch)
            elif ch == CR and opened_by_chinese:
                # 前瞻判断：后面跟 :,}] 则是结构性关闭，否则是内容
                j = i + 1
                while j < len(content) and content[j] in " \t\n\r":
                    j += 1
                if j >= len(content) or content[j] in ":,}]":
                    in_string = False
                    opened_by_chinese = False
                    result.append('"')
                else:
                    result.append(ch)
            else:
                # 字符串内其他字符（含内容中的中文引号），原样保留
                result.append(ch)
        else:
            if ch == '"':
                in_string = True
                opened_by_chinese = False
                result.append(ch)
            elif ch in (CL, CR):
                # 结构位置的中文引号，替换为标准双引号
                in_string = True
                opened_by_chinese = True
                result.append('"')
            elif ch == FC:
                result.append(":")
            else:
                result.append(ch)
        i += 1

    content = "".join(result)
    content = re.sub(r",\s*]", "]", content)
    content = re.sub(r",\s*}", "}", content)
    return content
```

状态机逻辑说明：

| 当前状态 | 遇到字符 | 处理 |
|---------|---------|------|
| 字符串外 | `"` | 进入字符串，标记为标准引号打开 |
| 字符串外 | `“` / `”` | 替换为 `"`，进入字符串，标记为中文引号打开 |
| 字符串外 | `：` | 替换为 `:` |
| 字符串内 | `\\x` | 转义序列，跳过两个字符 |
| 字符串内 | `"` | 关闭字符串 |
| 字符串内 | `”` + 由中文引号打开 | 前瞻判断：后接 `:,}]` 则关闭，否则保留为内容 |
| 字符串内 | 其他（含 `“`/`”`） | 原样保留（文本内容） |

## Bug: LLM 复述原文中的裸 ASCII 双引号导致 JSON 解析失败

### 影响范围

所有原文中包含 `\"SM-6\"`、`\"SM-2 防空导弹"` 等转义引号或混合引号的报文

### 问题现象

状态机修复（上一条）解决了中文引号的问题，但 index 5、index 6 等记录仍然存在 action 字段被污染的现象。

以 index 5 为例，原文内容：

```
...佐世保海军基地的"爱国者"防空系统，东海、南海的"SM-6"防空导弹保持战备状态
```

LLM 在 `original_text` 中复述时，`"爱国者"` 用了中文引号（状态机已能处理），但 `"SM-6"` 用了裸 ASCII 双引号 (U+0022)：

```
原文中: \"SM-6\"  (转义双引号, 源数据写法)
LLM 输出: "SM-6"  (裸ASCII双引号, 未转义)
```

LLM 输出的 JSON：
```json
"original_text": "...的"爱国者"防空系统，...的"SM-6"防空导弹保持战备状态"
                      ^^中文引号^^           ^^ASCII引号^^
                      U+201C/U+201D          U+0022 -- 和JSON结构引号相同！
```

JSON 解析器在 `的"` 处认为字符串结束，后面的 `SM-6"防空...` 变成非法 token，action 字段吞掉后续内容。

### 根因

这个问题和上一条不同：不是解析阶段的替换逻辑有问题，而是 LLM 输出了结构上非法的 JSON。根源在于源数据中的引号（转义引号 `\"`、中文引号、混合配对引号）被 LLM 复述时变成了裸 ASCII 双引号，与 JSON 结构引号冲突。

状态机无法解决这个问题，因为裸 ASCII `"` 在状态机中就是字符串边界，和真正的闭合引号完全相同，无法区分。

### 修复措施

采用输入预处理策略：在送入 LLM 之前清理文本中的所有引号，从源头消除 LLM 复述引号的可能。

新建 `llm_event_extractor_v3.py`，继承原始抽取器，覆盖预处理逻辑：

```python
class LLMEventExtractorV3(LLMEventExtractor):

    @staticmethod
    def preprocess_text(text: str) -> str:
        # Step 1: unescape \" to "
        text = text.replace('\\"', '"')

        # Step 2: normalize Chinese quotes to ASCII "
        text = text.replace('“', '"').replace('”', '"')

        # Step 3: remove paired "xxx" (1-30 chars inside), keep content
        text = re.sub(r'"([^"]{1,30})"', r'\1', text)

        # Step 4: remove any remaining stray "
        text = text.replace('"', '')

        return text
```

处理流程：

```
原文: ...的\"爱国者\"防空系统，...的\"SM-6\"防空导弹...
         ↓ Step 1: unescape \"
      ...的"爱国者"防空系统，...的"SM-6"防空导弹...
         ↓ Step 2: (此例无中文引号, 跳过)
      ...的"爱国者"防空系统，...的"SM-6"防空导弹...
         ↓ Step 3: remove paired "xxx"
      ...的爱国者防空系统，...的SM-6防空导弹...
         ↓ Step 4: (无残留, 跳过)
      ...的爱国者防空系统，...的SM-6防空导弹...   --> 送入 LLM
```

LLM 收到的文本无任何引号，`original_text` 中自然不会出现引号，JSON 解析不再出错。
