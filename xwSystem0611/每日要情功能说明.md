# 每日要情功能说明

## 功能概述

每日要情功能是一个基于后台作业的事件分析系统，通过定时任务自动分析报文并提取事件信息，支持五大主题的分类查询和手动触发分析。

**核心特性**：
- ✅ 后台定时任务：每日凌晨2点自动分析前一天新增报文
- ✅ 手动触发分析：支持自定义日期范围的手动分析
- ✅ 五大主题栏：美俄、印度、南亚、中亚、反恐
- ✅ 关键词库可配置：前端可见并支持手动编辑
- ✅ 秒级查询：分析结果存入数据库，前端秒级加载

---

## 架构设计

### 旧架构问题

```
前端 → 实时调用 /eventSplit → 等待10秒/篇 → 超时
```

**痛点**：
- 前端长时间等待（100篇报文需要15分钟）
- 请求超时（30秒限制）
- 用户体验差

### 新架构方案

```
┌─────────────┐     定时触发     ┌──────────────┐
│ 定时任务    │ ───────────────▶ │ 事件分析服务 │
│ (每日2点)   │                   │   (后台)     │
└─────────────┘                   └──────┬───────┘
                                         │
       ┌─────────────────────────────────┘
       │ 批量分析并存储
       ▼
┌──────────────┐
│ event_analysis │
│   数据表      │
└───────┬───────┘
        │
        │ 秒级查询
        ▼
┌──────────────┐
│   前端界面   │
│ 五大主题栏   │
└──────────────┘
```

**优势**：
- ✅ 前端查询秒级响应
- ✅ 后台异步处理，不阻塞用户
- ✅ 分析结果持久化存储
- ✅ 支持历史数据查询

## 数据库设计

### event_analysis 表

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INT | 主键ID（自增） |
| origin_text_id | INT | 原始报文ID（关联 origin_text.sid） |
| event_time | VARCHAR(100) | 事件时间 |
| event_location | VARCHAR(200) | 事件地点 |
| event_content | TEXT | 事件内容 |
| event_analysis | TEXT | 事件分析（6维分析） |
| analysis_date | DATE | 分析日期（用于日期范围查询） |
| create_time | DATETIME | 创建时间 |

**索引**：
- `idx_origin_text_id`：按报文ID查询
- `idx_analysis_date`：按日期范围查询
- `uk_origin_event`：联合唯一索引（避免重复分析）

```bash

# 查询事件
curl -X POST http://localhost:8081/api/eventAnalysis/query \
  -H "Content-Type: application/json" \
  -d '{"startDate":"2026-06-01","endDate":"2026-06-08","keywords":["美国","俄罗斯"]}'
```

## 定时任务

### 执行时间
```java
@Scheduled(cron = "0 0 2 * * ?")  // 每日凌晨2点
```

### 执行逻辑
1. 查询前一天新增的报文
2. 逐条调用算法服务 `/eventSplit` 接口
3. 解析返回的事件列表
4. 保存到 `event_analysis` 表
5. 记录执行日志

### 日志示例
```
2026-06-08 02:00:00 [INFO] ========== 开始每日事件分析任务 ==========
2026-06-08 02:00:00 [INFO] 开始分析日期范围: 2026-06-07 至 2026-06-07
2026-06-08 02:00:00 [INFO] 共查询到 15 条报文
2026-06-08 02:01:23 [INFO] 报文 9741 分析完成，识别 3 个事件
2026-06-08 02:05:45 [INFO] 分析完成: 总计=15, 成功=12, 失败=2, 跳过=1
2026-06-08 02:05:45 [INFO] ========== 每日事件分析任务结束 ==========
```

## 注意事项

1. **性能考虑**：
   - 每条报文分析耗时约10秒
   - 100条报文需要约15分钟
   - 建议在低峰期（凌晨）执行

2. **数据去重**：
   - 使用联合唯一索引 `uk_origin_event`
   - 避免重复分析同一报文的同一事件

3. **关键词匹配**：
   - 使用 OR 关系（命中任意一个即可）
   - 匹配字段：event_content、event_analysis、报文标题

4. **数据缓存**：
   - 关键词配置保存在 localStorage
   - 清除浏览器数据会重置为默认值

5. **定时任务监控**：
   - 日志路径：`后端/logs/application.log`
   - 搜索关键词："每日事件分析任务"

## 常见问题

### Q1: 为什么查询不到事件？
**A**: 可能原因：
1. 该日期范围内没有报文数据
2. 后台分析任务尚未执行
3. 关键词匹配不到任何事件

**解决方案**：
- 手动触发分析任务
- 调整关键词库
- 检查报文数据是否存在

### Q2: 手动触发分析后多久能看到结果？
**A**: 取决于报文数量：
- 10条报文：约2分钟
- 50条报文：约8分钟
- 100条报文：约15分钟

### Q3: 如何修改定时任务执行时间？
**A**: 修改 `DailyAnalysisTask.java` 中的 cron 表达式：
```java
@Scheduled(cron = "0 0 2 * * ?")  // 每日凌晨2点
// 改为：
@Scheduled(cron = "0 30 1 * * ?")  // 每日凌晨1:30
```

### Q4: 关键词配置会丢失吗？
**A**: 
- 保存在浏览器 localStorage 中
- 切换浏览器或清除数据会丢失
- 建议定期备份关键词配置
