# 算法代码说明

献微系统算法模块采用 Python Flask 2.0+ + OpenAI SDK 1.0+ 技术栈，提供事件抽取、报文融合、数据生成和事件拆分等核心算法服务。

---

## 一、项目概览

### 技术栈
- **Web 框架**: Flask 2.0+
- **LLM 调用**: OpenAI SDK 1.0+ (兼容多种 LLM API)
- **HTTP 客户端**: Requests
- **JSON 处理**: 标准库 json + 自定义解析器
- **日志**: logging
- **并发**: threading

### 目录结构
```
算法/
├── 算法服务/
│   └── app.py                          # Flask REST API（端口 5001）
├── 数据抽取-新/
│   ├── llm_event_extractor_v3.py       # V3 事件抽取器（主入口，直接抽取+分类）
│   ├── llm_event_extractor.py          # LLM 事件抽取（底层精确抽取）
│   ├── classify_client.py              # RexUniNlu 分类客户端
│   ├── config.json                     # 配置文件（LLM/分类服务地址）
│   └── prompt_templates.py             # Prompt 模板
├── 报文融合/
│   ├── fusion_extractor.py             # 报文融合算法
│   └── fusion_config.json              # 融合配置
├── 数据生成/
│   ├── data_generator.py               # LLM 数据生成
│   ├── candidate_pools.py              # 候选池管理
│   └── examples/                       # 样例数据
└── 事件拆分/
    └── event_splitter.py               # 事件原子化拆分
```

## 三、事件抽取模块

### 3.1 V3 事件抽取器 - [llm_event_extractor_v3.py](../算法/数据抽取-新/llm_event_extractor_v3.py)

**职责**：从报文中提取事件，LLM 直接输出分类字段

**核心改进**：
1. **输入预处理**：防御引号问题（替换特殊引号为标准引号）
2. **Prompt 优化**：LLM 直接输出分类字段（船只、飞机、武器等），无需二次分类
3. **三重 JSON 解析**：直接解析 → Markdown 代码块提取 → 花括号计数修复
4. **状态机修复**：处理不完整 JSON

#### 3.1.1 抽取流程

```
输入文本
  ↓ 段落分割（按句号、感叹号、问号、换行符）
  ↓ 输入预处理（引号标准化）
  ↓ 构造 Prompt（含分类字段要求）
  ↓ 调用 LLM API
  ↓ 三重 JSON 解析
  ↓ 后处理（生成 subject/labels）
  ↓ 返回事件列表
```

#### 3.1.2 Prompt 模板（简化版）

```python
def parse_llm_response(response_text):
    # 策略1：直接解析
    try:
        return json.loads(response_text)

    # 策略2：提取 Markdown 代码块
    import re
    match = re.search(r'```json\s*(.*?)\s*```', response_text, re.DOTALL)
    if match:
        try:
            return json.loads(match.group(1))
        except:
            pass
    
    # 策略3：花括号计数修复
    try:
        return fix_incomplete_json(response_text)

        pass
```

## 四、报文融合模块

### 4.1 融合抽取器 - [fusion_extractor.py](../算法/报文融合/fusion_extractor.py)

**职责**：多篇报文融合生成综合报告

#### 4.1.1 融合流程

```
Step 1: 数据预处理
  ↓ 统一字段格式（兼容新旧版本）
  ↓ 提取时间线、实体、标签

Step 2: 信息整合（代码实现）
  ↓ 合并所有事件的时间线（按时间排序）
  ↓ 合并所有实体（智能归一化）
  ↓ 合并所有标签（去重）

Step 3: LLM 生成
  ↓ 生成融合标题
  ↓ 生成摘要
  ↓ 生成详细内容（Markdown 格式）

Step 4: 结果组装
  ↓ 构建目标列表表格（entities 字段）
  ↓ 构建时间线（timeline 字段）
  ↓ 输出完整融合报告
```

#### 4.1.2 关键函数

**数据预处理**：
```python
def preprocess_reports(reports):
    """统一字段格式"""
    for report in reports:
        # 确保 extractionResult 存在
        if 'extractionResult' not in report:
            report['extractionResult'] = {
                'events': [],
                'labels': [],
                'entities': {}
            }
        
        # 解析 events_json（兼容字符串格式）
        events = report['extractionResult'].get('events')
        if isinstance(events, str):
            events = json.loads(events)
        report['extractionResult']['events'] = events
    
    return reports
```

**实体智能归一化**：
```python
def normalize_entity_name(name):
    """
    实体归一化：剥离前后缀 + 标点统一
    
    示例：
    - "美国总统拜登" → "拜登"
    - "第七舰队。" → "第七舰队"
    - "萨德反导系统" → "萨德反导系统"
    """
    # 剥离常见前缀
    prefixes = ['美国总统', '俄罗斯总统', '中国', '美国', '日本']
    for prefix in prefixes:
        if name.startswith(prefix):
            name = name[len(prefix):]
    
    # 剥离标点
    name = name.strip('。，、！？：；')
    
    return name

def merge_entities(all_events):
    """合并实体，按分类聚合，归一化同义实体"""
    entities_dict = {}  # {category: {normalized_name: [actions]}}
    
    for event in all_events:
        for category in ['人物', '组织', '船只', '武器', '装备']:
            if category in event and event[category]:
                for entity_name in event[category]:
                    normalized = normalize_entity_name(entity_name)
                    action = f"{event['time']}: {event['original_text']}"
                    
                    if category not in entities_dict:
                        entities_dict[category] = {}
                    if normalized not in entities_dict[category]:
                        entities_dict[category][normalized] = []
                    entities_dict[category][normalized].append(action)
    
    # 转换为目标列表格式
    entities_list = []
    for category, items in entities_dict.items():
        for name, actions in items.items():
            entities_list.append({
                'category': category,
                'name': name,
                'action': '\n'.join(actions)
            })
    
    return entities_list
```

**LLM 生成融合内容**：
```python
def generate_fusion_content(all_events, timeline, entities, labels):
    """调用 LLM 生成融合报告的标题、摘要和详细内容"""
    
    # 构造 Prompt
    prompt = f"""
你是一个军事情报分析专家。根据以下多篇报文的事件信息，生成一份综合分析报告。

时间线：
{json.dumps(timeline, ensure_ascii=False, indent=2)}

目标列表：
{json.dumps(entities, ensure_ascii=False, indent=2)}

标签：{', '.join(labels)}

请生成：
1. 报告标题（简洁，20字以内）
2. 摘要（200字左右）
3. 详细内容（Markdown 格式，包含态势分析、行动研判、趋势预测）

输出格式为 JSON：
{{
  "title": "标题",
  "summary": "摘要",
  "content": "详细内容"
}}
"""
    
    # 调用 LLM
    response = client.chat.completions.create(
        model=config['llm']['model'],
        messages=[{"role": "user", "content": prompt}],
        temperature=0.3,
        max_tokens=8000
    )
    
    result = json.loads(response.choices[0].message.content)
    return result['title'], result['summary'], result['content']
```

### 6.1 事件拆分器 - [event_splitter.py](../算法/事件拆分/event_splitter.py)

**职责**：将复杂事件拆分为原子事件

**拆分规则**：
- 一个事件包含多个主体 → 拆分为多个事件（每个事件一个主体）
- 一个事件包含多个地点 → 拆分为多个事件（每个事件一个地点）
- 一个事件包含多个行动 → 拆分为多个事件（每个事件一个行动）

**示例**：
```
输入事件：
{
  "subject": ["里根号航母", "卡尔文森号"],
  "location": ["横须贺", "关岛"],
  "action": "部署并保持战备状态"
}

输出事件：
[
  {"subject": "里根号航母", "location": "横须贺", "action": "部署并保持战备状态"},
  {"subject": "里根号航母", "location": "关岛", "action": "部署并保持战备状态"},
```

## 十、性能优化

### 10.1 V3 抽取器优化

1. **输入预处理**：防御引号问题，减少 JSON 解析失败率
2. **三重 JSON 解析**：提高解析成功率（从 70% 提升至 95%）
3. **状态机修复**：处理不完整 JSON，避免完全失败
4. **缓存机制**：相同文本缓存抽取结果（`llm_calls_saved` 计数）

### 10.2 融合算法优化

1. **实体归一化**：智能合并同义实体（"美国总统拜登" = "拜登"）
2. **批量处理**：一次 LLM 调用生成标题+摘要+内容，减少调用次数
3. **时间线排序**：按时间自动排序，提升报告可读性

## 十二、注意事项

1. **LLM 调用超时**：抽取和融合接口设置 20 分钟超时，适配大模型处理时间
2. **GLM-5 特殊处理**：GLM-5 模型返回内容在 `reasoning_content` 字段，需特殊解析
3. **配置热更新**：修改 `config.json` 后需完全重启容器（停止 → 删除 → 启动）
4. **JSON 解析容错**：V3 抽取器采用三重解析策略，提高鲁棒性
5. **实体归一化**：融合算法内置实体归一化逻辑，自动合并同义词
6. **原文保留**：融合报告的 `timeline[].description` 和 `entities[].action` 均取自 `event.original_text`，比 `action` 字段更完整
