# 献微系统问题汇总

## 问题1：前端加载缓慢

### 发现时间
2026-04-19

### 问题描述
前端登录后，"信息来源"下的类别和每个类别（如"军事演习")的内容出现很慢，页面加载延迟明显。

### 问题定位

#### 后端SQL无分页
- `UygurMapper.getTextListAll()` 查询全部 **19543条** 记录，无 LIMIT 分页
- `UygurMapper.getTextListByType()` 按分类查询，也无分页
- `UygurMapper.getTextListByModalType()` 按模态查询，也无分页

#### 参数未使用
- 前端传递 `pageNum` 和 `pageSize` 参数
- 后端 `GetListRequest` 接收参数但 **未传递给Mapper**
- `UygurServiceImpl.getTextList()` 直接调用无分页的Mapper方法

#### 表关联增加查询成本
```sql
select o.*, e.labels_json as labelsJson 
from origin_text o 
left join extraction_result e on o.sid = e.origin_text_id 
order by o.sid
```
- LEFT JOIN 19543条记录 + extraction_result表
- 无索引优化，全表扫描

### 影响范围
- 前端列表加载慢
- 选择分类后加载慢
- 分页切换无效（实际返回全部数据）

### 修复方案
1. 修改Mapper SQL添加分页：`LIMIT #{pageSize} OFFSET #{offset}`
2. 修改Service计算offset：`(pageNum - 1) * pageSize`
3. 返回分页数据结构：`{total, list}`

### 修复优先级
**高** - 影响用户体验，页面加载时间过长

### 状态
**已解决** - 后端添加数据库分页查询，Mapper新增分页方法，Service实现分页逻辑

### 修改文件
- `后端/src/main/java/com/qy/dch/mapper/UygurMapper.java` - 新增分页查询方法和计数方法
- `后端/src/main/java/com/qy/dch/service/UygurService.java` - 新增getTextListPaged接口
- `后端/src/main/java/com/qy/dch/service/impl/UygurServiceImpl.java` - 实现分页查询逻辑
- `后端/src/main/java/com/qy/dch/controller/UygurController.java` - 调用分页方法
- `后端/src/main/java/com/qy/dch/dto/PageResultDTO.java` - 新增分页结果DTO

### 相关文件
- `后端/src/main/java/com/qy/dch/mapper/UygurMapper.java` (第48-72行)
- `后端/src/main/java/com/qy/dch/service/impl/UygurServiceImpl.java` (第101-117行)
- `后端/src/main/java/com/qy/dch/request/GetListRequest.java`
- `前端/qyportrait/src/components/searchResult.vue` (第952-960行)

## 问题4：融合报告保存失败

### 发现时间
2026-04-19

### 问题描述
融合完成后，点击"保存报告"按钮保存失败，后端返回JSON反序列化错误。

### 问题定位

#### 后端日志错误
```
Cannot deserialize value of type `java.lang.String` from Array value (token `JsonToken.START_ARRAY`)
```

#### 类型不匹配
| 字段 | 前端发送类型 | 后端FusionDTO类型 | 问题 |
|------|-------------|-------------------|------|
| timeline | Array `[]` | String | ❌ 无法将数组反序列化为字符串 |
| entities | Object `{}` | String | ❌ 无法将对象反序列化为字符串 |
| labels | Array `[]` | String | ❌ 无法将数组反序列化为字符串 |

#### 代码位置
- 前端 `fusionResult.vue` 第349行：直接发送 `this.fusionData` 对象
- 后端 `FusionDTO.java`：timeline、entities、labels 定义为 String 类型

### 影响范围
- 融合报告无法保存到数据库
- 用户编辑的报告内容丢失

### 修复方案
**方案1：前端修复**
```javascript
// 保存前将数组/对象转为JSON字符串
const saveData = {
  ...this.fusionData,
  timeline: JSON.stringify(this.fusionData.timeline),
  entities: JSON.stringify(this.fusionData.entities),
  labels: JSON.stringify(this.fusionData.labels)
};
request.post('/api/fusion/save', saveData);
```

**方案2：后端修复**
```java
// FusionDTO 使用 List 和 Map 类型接收，保存时转为JSON字符串
private List<Map<String, Object>> timeline;
private Map<String, Object> entities;
private List<String> labels;
```

### 修复优先级
**高** - 核心功能无法使用

### 状态
**已解决** - 前端保存前将数组/对象转为JSON字符串，与后端FusionDTO类型匹配

### 相关文件
- `前端/qyportrait/src/components/fusionResult.vue` (第346-364行)

## 问题5：融合内容显示不全

### 发现时间
2026-04-19

### 问题描述
融合报告的"详细内容"部分只显示了开头一段文字，后面的内容未显示。似乎是换行符 `\n` 未被正确渲染。

### 问题定位

#### 后端返回格式
- 融合内容为 Markdown 格式，包含 `\n` 换行符
- 例如：`"## 一、事件概述\n\n本次融合报告涵盖..."`

#### 前端渲染方式
```vue
<div v-if="fusionData.content" v-html="fusionData.content"></div>
```
- `v-html` 不会自动将 `\n` 转换为 `<br>`
- HTML中换行符默认被忽略，需要CSS或标签处理

#### Markdown未解析
- 后端返回的是原始 Markdown 文本，而非 HTML
- 前端直接渲染为 HTML，Markdown 标记如 `##`、`**` 等未被解析

### 影响范围
- 融合报告详细内容显示不完整
- 用户无法看到完整的分析报告

### 修复方案
**最终方案：前端解析Markdown**
```javascript
// 安装marked库
npm install marked --save

// 导入并使用marked解析Markdown
import { marked } from 'marked'

// 添加计算属性
computed: {
  contentHtml() {
    if (this.fusionData.content) {
      return marked(this.fusionData.content);
    }
    return '';
  }
}

// 模板中使用计算属性
<div v-html="contentHtml"></div>
```

配合CSS样式：
```css
.detail-content {
  white-space: pre-wrap;
  line-height: 1.6;
}
```

### 修复优先级
**中** - 影响报告内容展示

### 状态
**已解决** - 前端使用marked库解析Markdown内容为HTML，正确渲染标题、段落、换行

### 相关文件
- `前端/qyportrait/src/components/fusionResult.vue` (第91-97行、computed属性、marked导入)
- `算法/报文融合/fusion_extractor.py` (Markdown生成部分)

## 问题6：融合报告标签包含固定默认标签

### 发现时间
2026-04-19

### 问题描述
融合报告的标签中出现了未经过智能分类的固定默认标签（如"军事"、"国际"、"政治"等），与智能分类标签混在一起，导致标签不纯粹。

例如：
- 报文9741智能分类标签: `['部队', '武器']`
- 报文9742智能分类标签: `['部队']`
- 融合后标签: `['部队', '武器', '军事', '国际', '政治']` ← 混入了默认标签

### 问题定位

#### 融合标签生成流程
1. **抽取阶段**: RexUniNlu智能分类，生成标签如 `['部队', '武器']`
2. **融合阶段**: 合并各报文标签，但强制补充默认标签

#### 代码问题
文件: `算法/报文融合/fusion_extractor.py` (第295-305行)

```python
def _merge_labels(self, all_labels: List[str]) -> List[str]:
    """合并标签（代码实现）"""
    labels = list(set(all_labels))  # 去重合并

    # ⚠️ 问题：强制补充固定默认标签
    default_labels = ['军事', '国际', '政治', '外交', '安全']
    while len(labels) < 5 and default_labels:
        label = default_labels.pop(0)
        if label not in labels:
            labels.append(label)

    return labels[:8]
```

#### 标签来源对比
| 标签类型 | 来源 | 是否智能分类 |
|---------|------|-------------|
| 部队、武器、军舰 | RexUniNlu | ✅ 智能分类 |
| 军事、国际、政治 | 硬编码默认值 | ❌ 固定标签 |

### 影响范围
- 融合报告标签不准确
- 用户无法区分智能标签和固定标签
- 标签分类功能失去意义

### 修复方案
修改 `_merge_labels()` 方法，只使用智能分类标签，不补充默认标签：

```python
def _merge_labels(self, all_labels: List[str]) -> List[str]:
    """合并标签（仅使用智能分类标签）"""
    # 去重合并，不补充默认标签
    labels = list(set(all_labels))
    # 按出现频率排序，取前8个
    return labels[:8]
```

### 修复优先级
**中** - 影响标签准确性

### 状态
**已解决** - 融合标签仅使用智能分类结果，不再补充默认标签

### 相关文件
- `算法/报文融合/fusion_extractor.py` (第295-305行)

## 问题7：融合请求前端一直显示"正在生成..."

### 发现时间
2026-04-19

### 问题描述
前端发起融合请求后，页面一直显示"正在生成..."，长时间不显示结果，即使后端已完成融合处理。

### 问题定位

#### 后端日志分析
通过分析后端日志发现请求混乱现象：

| 时间 | 线程 | 事件 |
|------|------|------|
| 20:59:37 | `nio-8081-exec-5` | 发送融合请求 |
| 20:59:37 | 算法服务 | 开始LLM处理 |
| **21:01:04** | `io-8081-exec-10` | **发送新请求（打断旧请求）** |
| 21:01:53 | 算法服务 | HTTP 200响应（给新请求） |
| 21:02:51 | `io-8081-exec-10` | 收到响应 |

#### 原因分析
1. **前端缺少防重复提交机制**
   - 用户多次点击融合按钮
   - `fusing`变量在finally块才设置false，第一次请求未完成时第二次请求已发送

2. **并发请求导致响应混乱**
   - 算法服务收到新请求后，原请求响应丢失
   - 原请求线程`nio-8081-exec-5`的axios一直pending

3. **请求超时问题**
   - 后端readTimeout=600000（10分钟）
   - 前端axios timeout=1000000ms（16分钟）
   - 请求被新请求打断，无法正确返回

### 影响范围
- 融合功能无法正常使用
- 用户操作体验差，页面卡住
- 后端资源浪费

### 修复方案

**方案1：前端添加防重复提交**
```javascript
// fusionResult.vue startFusion()方法开头添加
async startFusion() {
  // 防止重复提交
  if (this.fusing) {
    this.$message.warning('正在融合中，请勿重复操作');
    return;
  }
  // ... 其他代码
}
```

**方案2：添加请求取消机制**
```javascript
// 保存请求引用，新请求时取消旧请求
let fusionRequest = null;
if (fusionRequest) {
  fusionRequest.cancel('取消旧请求');
}
fusionRequest = axios.post('/api/fusion/create', ...);
```

### 修复优先级
**高** - 核心功能无法正常使用

### 状态
**已解决** - 前端startFusion()方法添加fusing状态检查，防止重复提交

### 相关文件
- `前端/qyportrait/src/components/fusionResult.vue` (startFusion方法)

## 问题8：融合请求前端超时导致无法接收结果

### 发现时间
2026-04-20

### 问题描述
前端发起融合请求后，页面一直显示"正在生成..."，即使后端成功完成融合处理并返回结果，前端也无法接收显示。

### 问题定位

#### 后端处理时间线
```
09:38:55 - 收到融合请求
09:57:42 - 抽取完成 (耗时约19分钟)
09:59:22 - 融合完成并返回结果
```

#### 超时配置对比
| 配置项 | 超时时间 | 说明 |
|--------|---------|------|
| 前端 axios timeout | 1000000ms (16.7分钟) | ❌ 不足 |
| 后端 readTimeout | 600000ms (10分钟) | 算法服务调用 |
| 实际处理时间 | 约20分钟 | 抽取+融合 |

#### 根本原因
前端axios超时时间(16.7分钟)小于后端实际处理时间(约20分钟)。请求在16.7分钟后超时断开，但后端仍在处理，导致前端无法接收最终结果。

### 影响范围
- 融合功能无法正常使用
- 前端显示"融合失败"或一直pending
- 用户需要重新发起请求

### 修复方案
修改前端axios超时配置，从16.7分钟增加到30分钟：
```javascript
// request.js
timeout: 1800000  // 30分钟
```

### 修复优先级
**高** - 核心功能无法正常使用

### 状态
**已解决** - 前端axios超时时间增加到30分钟

### 相关文件
- `前端/qyportrait/src/utils/request.js` (第34行)
